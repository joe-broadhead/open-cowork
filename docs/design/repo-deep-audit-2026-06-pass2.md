# Open Cowork — Deep Repository Audit, Pass 2 (2026-06-24)

**Method.** Twelve parallel dimension-audits (auth/tenant isolation, secrets/crypto, injection/SSRF, Electron/web security, cloud DB scalability, cloud runtime scalability, desktop/runtime-host performance, gateway scalability, correctness/concurrency/store-parity, code quality/dead code, docs drift, production-readiness/ops/testing). Each auditor read the actual source and was briefed on the **first pass's ~20 fixes** so it hunted only for what was missed or is deeper. The three flagship cross-cutting findings (ReDoS, `/readyz` disclosure, desktop per-event fsync) were independently re-verified by hand.

**Severity scale (repo-wide, normalized):** **P0** = outage/data-loss under normal use or growth, or an actively exploitable hole. **P1** = serious; incident under load or specific conditions; meaningful security/correctness gap. **P2** = should-fix; degradation / latent bug / defense-in-depth. **P3** = polish.

**Headline.** Tenant isolation, token crypto, envelope encryption, Electron hardening, and the supply chain are **genuinely strong** (the auth audit found no P0/P1; isolation is enforced from the principal, role re-derived from the stored membership). The real risk is concentrated in five themes: (1) a **verified ReDoS** in the shared log sanitizer; (2) **cloud SSE-polling + unbounded-read scaling cliffs** with no `statement_timeout`; (3) **desktop main-thread stalls on cloud workspaces** (per-event fsync + SQLite reindex); (4) a **silent gateway delivery blackhole** that keeps `/ready` green; (5) **retention/ops gaps** (unbounded core tables, missing probes/preStop, dark object-store metrics). Nothing here is an active cross-tenant data leak or RCE.

Counts: **P0 ×4, P1 ×18, P2 ×17, P3 ×16**, plus store-parity divergences, a dead-code shortlist, and doc drift.

## P0 — Critical (falls over under normal use/load)

### [P0-1] Cloud SSE replay-hub polls ~6 control-plane queries per connection per second, unconditionally — *cloud, scalability*
`sse-replay.ts:102`, `http-server.ts:260-263,591-598,688-695`, `session-service.ts:1052-1074`
Every session-events SSE connection registers its own `setInterval(pollMs=1000)`; each poll runs `service.listEvents` → `getSessionView` (`ensurePrincipal`→`resolvePrincipalMembership` + `getSession` + `getSessionProjection`) **then** `listSessionEvents` (`requireSession`→`requireTenant` + a `cloud_sessions` SELECT + a `cloud_session_events` SELECT) ≈ **6 queries/sec/connection, even when idle**. The in-memory `eventBus` already carries live events; the poll only covers cross-replica/retention gaps yet runs at full cadence.
**Impact:** 1,000 idle SSE clients ⇒ ~6,000 control-plane queries/sec of pure polling, independent of event volume; the pg pool (`max=10`) saturates long before any real work. Connection count becomes the dominant DB load.
**Fix:** Make the poll demand-driven — skip when the topic's `lastSequence` already equals a cached `max(sequence)`; authorize once at connect (drop `getSessionView` from the per-poll path); collapse identical session/workspace topics to one shared poll; lengthen the fallback interval (15–30s) while the live bus carries the hot path. *Confidence: high (specific trace); load-test to quantify.*

### [P0-2] Unbounded event-log reads + no `statement_timeout` → pool exhaustion from one big session — *cloud, scalability*
`session-projection-service.ts:147`, `session-service.ts:1082`, `postgres-control-plane-store.ts:1357-1369`, `postgres-pool-options.ts:38`
`repairSessionProjection` and `getSessionProjectionStatus` (a plain authenticated GET, `http-server.ts:1223`) call `listSessionEvents(tenant, session, 0)` with **no limit** — the whole `cloud_session_events` history is materialized into one array. The initial SSE connect drains the full history too (`afterSequence` defaults to 0; the `SSE_REPLAY_BATCH=1000` cap is only on the *poll* path). The pool ships `statement_timeout = 0` (disabled) and `withTransaction` sets no per-tx timeout, so any of these can run unbounded and pin a connection out of a pool of 10.
**Impact:** A single long-lived session (thousands–millions of events) turns a small GET / SSE connect / projection rebuild into an unbounded read that exhausts the pool → cascading 503s. `cloud_session_events` has **no retention** (see P1-7), so this only worsens over time.
**Fix:** Keyset-page the rebuild and bound the initial SSE drain with `SSE_REPLAY_BATCH`; replace `projection-status` with `count(*)`/`max(sequence)` aggregates; ship a non-zero default `statement_timeout` (30–60s) with DDL/migration exempt. *Confidence: high.*

### [P0-3] Desktop cloud cache re-serializes + fsyncs the entire transcript synchronously on every streamed cloud event — *desktop, performance* **(verified by hand)**
`cloud-workspace-adapter.ts:733` → `cloud-workspace-cache.ts:495-541` → `fs-atomic.ts:80`
`subscribeSessionEvents.onEvent` calls `cache.setEventCursor(...)` synchronously per event, which does `readFileSync` + `decryptString` + `JSON.parse`, then `JSON.stringify` + `encryptString` + `writeFileAtomic` (temp→**fsync**→rename). In `full` mode (the default, `config-types.ts:624`) the serialized blob includes every cloud session's full `SessionView` transcript. I confirmed `onEvent` fires per transport event with no debounce and no `await`.
**Impact:** Every cloud session event blocks the Electron main loop on read-whole-file + decrypt + stringify + encrypt + fsync of *all* cloud transcripts — cost scales with total transcript size, paid per event. Renders cloud workspaces janky-to-unusable during generation.
**Fix:** Keep the latest sequence per stream in memory and debounce/coalesce the durable write (flush on idle/interval/close); store cursors in a tiny dedicated file so a cursor bump never re-serializes transcripts. *Confidence: high.*

### [P0-4] Desktop thread-index reindexes per event: synchronous multi-statement SQLite txn + 3× `chmod` + full query-cache wipe, per active session every 750ms — *desktop, performance*
`session-event-dispatcher.ts:579` → `thread-index-service.ts:307-315` → `thread-index-store.ts:138-264,145,158-162`
`dispatchRuntimeSessionEvent` schedules a metadata refresh for **every** event (incl. high-frequency `text`/`reasoning`); the 750ms debounce is **per session**, so N streaming sub-agents each fire independently. Each fire runs a 29-column upsert + `replaceAgents`/`replaceTools`/`replaceSuggested` (DELETE+re-INSERT) inside a `withTransaction` that **also** `chmod`s the db/-wal/-shm and **clears the entire search + facet cache** on every commit. `node:sqlite` is synchronous on the main thread.
**Impact:** A streaming session forces a synchronous multi-statement SQLite transaction + 3 chmod syscalls at least every 750ms × active-session count, and the Studio search/facet caches are permanently cold while anything streams.
**Fix:** Skip the write when derived metadata is unchanged (compare a revision/hash); recompute suggestions only on terminal events; hoist `ensureThreadIndexDbFileModes` out of the per-tx path; invalidate the query cache granularly per session, not clear-all.

## P1 — High

### Security
- **[P1-S1] ReDoS in the shared log sanitizer (un-truncated input) — DoS** *(found by 2 auditors; reproduced by hand)* — `packages/shared/src/log-sanitizer.ts:48-49`, `node/logger.ts:152`, `observability.ts:121,142`, `audit-redaction.ts:12`. `EMAIL_PATTERN` and `KEYED_SECRET_PATTERN` have overlapping quantifiers; `sanitizeLogMessage` runs them on the **full** string (the `.slice()` in observability/audit happens *after* the regex; the logger path has no cap at all). Measured O(n²): EMAIL 49→190→**753 ms** at 10/20/40 KB (≈ minutes at ~500 KB). Reachable via raw `error.message` logged unbounded (`recordLoopError` `app.ts:880,903`; HTTP-500 logger `http-server.ts:1735,1743`). **Fix:** truncate input at the top of `sanitizeLogMessage` (≤ a few KB) **and** de-ambiguate both patterns; fix the duplicate EMAIL regex at `observability.ts:142`. *High.*
- **[P1-S2] Unauthenticated `/readyz` leaks raw object-store / DB driver error text** *(verified by hand)* — `readiness.ts:54-63`, `http-server.ts:1611-1624`. `errorDetail(error).slice(0,512)` puts the raw message into the report; `/readyz` serializes it **before** `enforceIpRateLimit`/auth. Sources are not static: `object-store.ts:223` interpolates up to 512 chars of the raw GCS/Azure HTTP body; the S3 adapter re-throws the raw AWS exception (which echoes `AWSAccessKeyId` on `SignatureDoesNotMatch`); `listSchemaMigrations` surfaces raw `pg` errors (host/role/conn-string). **Fix:** route `detail` through `sanitizeForExport`, or store a static string and log the raw detail server-side only. *High.*
- **[P1-S3] Webhook SSRF re-opens cloud-metadata (IMDS) when `allowPrivateDelivery` is enabled** — `webhook-url-policy.ts:45,159-168` vs canonical `mcp-url-policy.ts:184,225`. The canonical policy blocks `cloud-metadata` (`169.254.169.254`, `metadata.google.internal`) **even when private network is allowed**; the webhook policy folds IMDS into the generic private-IP rule and gates that whole rule behind `!allowPrivateDelivery`. So an operator who enables `OPEN_COWORK_GATEWAY_WEBHOOK_ALLOW_PRIVATE_DELIVERY` (a supported config) silently re-exposes IMDS credential exfil for webhook destinations. **Fix:** classify cloud-metadata separately and block it unconditionally; ideally consume the `mcp-url-policy` primitives (the desktop broker policy already reuses them). *High.*

### Cloud scalability / DB
- **[P1-C1] No per-org cap on concurrent SSE connections** — `http-server.ts:1638,1702`, `session-service.ts:2308-2326`. The rate limiter charges an SSE connect as one request, then the connection lives indefinitely; `maxConcurrentSessionsPerOrg` gates *creation*, not connections. One org can open thousands of long-lived streams within its 600/min budget — each a socket + timers + bus subscription + (P0-1) DB poll. **Fix:** per-org/per-token concurrent-SSE ceiling enforced at connect in `CloudSseStreamRegistry`. *High.*
- **[P1-C2] `cloud_rate_limits` and `cloud_auth_backoff` grow unbounded (one row per source IP, forever)** — `postgres-store-domains/rate-limits.ts:32-46`. Upsert-only, never pruned; retention covers only channel tables. A public bind seeing many client IPs accrues one permanent row per IP, degrading the per-request upsert that runs on **every** HTTP request. **Fix:** add a bounded prune (`window_started_at_ms < now - windowMs`) to the scheduler sweep. *High.*
- **[P1-C3] `cloud_session_events` / `cloud_audit_events` / `cloud_usage_events` have no retention** *(found by 2 auditors)* — `scheduler.ts:6-26` (comment: "audit, usage, session/workspace events…intentionally excluded"). The SSE replay log grows without bound on any active tenant → degrades claim/replay queries + disk. Combined with P0-2's unbounded reads, these are the long-term cliff. **Fix:** bounded retention/compaction sweep for `cloud_session_events` (keep last-N per session or events for terminal sessions older than N days), wired into the existing scheduler loop; document the knob. *High.*
- **[P1-C4] Missing indexes on scheduled/hot paths** (all verified against the index inventory): scheduler `claimDueWorkflowRun` is a cross-tenant scan+sort (`reaper_idx` is partial `WHERE claim_token IS NOT NULL`, the opposite predicate) — `workflows.ts:246-270`; channel-delivery retention sweep full-scans+sorts the hottest table (no `(status, updated_at)` index) — `channel-deliveries.ts:228-243`; expired-interaction sweep has no `expires_at` index — `postgres-control-plane-store.ts:948-957`; webhook replay-cache trim full-scans + `OFFSET`-sorts `cloud_webhook_replay_claims` (no `seen_at_ms` index) on **every** inbound webhook — `webhooks.ts:107-126`; `listMembershipsForAccount` full-scans `cloud_memberships` (no `account_id`-leading index) on the principal-resolution path — `identity.ts:232-238`; active-worker quota is a 3-table `COUNT(*)` join with a non-sargable `coalesce()` (no gauge was added for worker leases) — `quotas.ts:108-117`. **Fix:** add the partial/secondary indexes named in the DB report; extend the O(1) gauge pattern to worker-lease concurrency. *High.*
- **[P1-C5] Scheduler claim loop is unbounded and writes O(K²) heartbeat bytes** — `scheduler.ts:77-88`, `worker-heartbeats.ts:26-47`. `while(true){ claim; activeSessionIds.push; recordWorkerHeartbeat({activeSessionIds}) }` — the K-th claim serializes a K-element JSON array ⇒ quadratic write growth and K upserts per tick, draining the whole backlog inline and starving the reaper/retention. **Fix:** cap claims per tick; write the heartbeat once per tick. *High.*

### Desktop / runtime-host performance
- **[P1-D1] `JSON.stringify(message)` over the whole transcript per streamed patch (renderer)** — `ChatView.tsx:195`. `stabilizeMessage` JSON-stringifies **every** message each patch (~30×/s during generation), including settled ones; virtualization culls DOM but not this memo. **Fix:** cheap structural signature (content length + reasoning/segments/attachments lengths + order + role), or short-circuit messages below the streaming message's order. *High.*
- **[P1-D2] `SessionInspector` JSON-serializes every tool input/output in the render body, unmemoized, open by default** — `SessionInspector.tsx:272-282`. During generation it stringifies every tool call's input+output across the session and all sub-agent task runs per patch, just to feed a token-estimate bar. **Fix:** `useMemo` the payloads/breakdown; use `String(x).length` (the estimate is `/4`). *High.*
- **[P1-D3] `listLocalArtifactIndex` materializes a full `SessionView` per session and re-runs `listCoordinationTasks(limit:1000)` inside the per-session loop** — `artifact-index.ts:418-453`. O(sessions) view derivations + O(sessions) redundant coordination queries; `updateLocalArtifactStatus` does this just to find one artifact. **Fix:** hoist the coordination query out of the loop; filter at source when `sessionId` is given. *High.*

### Gateway
- **[P1-G1] Clean SSE close silently blackholes the cloud→channel delivery pipe; `/ready` stays green** — `gateway-runtime.ts:80-97`, `cloud-client/.../transport.ts:289-319`. The delivery subscription opens once; the fetch-SSE transport treats a graceful stream end as `chunk.done → break` and **never calls `onError`**, so `cloudDeliveriesHealthy` stays `true` when the cloud/LB closes an idle connection (deploy, idle timeout, scale event). The pass-1 `/ready` gate only catches the `onError` path. There is no heartbeat/watchdog/resubscribe. **Impact:** silent, indefinite delivery outage the orchestrator can't detect. **Fix:** surface clean-close as a terminal/`onComplete` signal; auto-resubscribe with backoff (the session-stream-manager already does this) and add a last-delivery watchdog. *High.*
- **[P1-G2] No concurrency cap / backpressure on the delivery path → retry & throughput storm on backlog drain** — `gateway-runtime.ts:83-90,260-310`. `onDelivery` fires `handleDelivery` immediately; `inFlightDeliveries` is only a shutdown-drain set, never a semaphore. A queued backlog (e.g. after downtime) spawns unbounded concurrent outbound sends → provider 429s → transient-retry storm → out-of-order channel messages. The session-event path got `maxQueueDepth` in pass 1; the delivery path did not. **Fix:** bounded worker pool + SSE backpressure; serialize per `channelBindingId`/target for ordering. *High.*
- **[P1-G3] Standalone `/ready` runs an unbounded full-table identity scan + OpenCode probe, unauthenticated, on every probe** — `standalone-gateway/server.ts:138-142`, `doctor.ts:94`, `postgres-repository.ts:385-399`. `/ready` runs the full doctor unconditionally; `identityAuthorizationSummary` does `SELECT * FROM standalone_gateway_channel_identities` (no LIMIT) + materializes all rows + an OpenCode `/health` round-trip. An orchestrator (or anonymous attacker) hammering `/ready` forces repeated full scans. **Fix:** aggregate SQL for the summary; cache the doctor result behind a short TTL; move the full doctor behind admin auth. *High.*
- **[P1-G4] Standalone single-writer lease does NOT stop jobs on lease loss (split-brain window)** — `standalone-gateway/main.ts:101-122`, `runtime.ts:114-133`. `runDueJobs`/`claimNextJob` and `handleMessage` have **zero** lease awareness; lease loss only triggers an async `process.exit(1)` on the next 10s renewal tick. Between losing the lease and exiting, a new daemon can acquire it while the old keeps claiming/finishing jobs and prompting OpenCode → duplicate execution. (Retention *is* lease-guarded in the same transaction — the jobs path is the asymmetric gap.) **Fix:** make `claimNextJob` lease-aware (claim+verify in one tx, like `pruneRetention`); stop the maintenance/provider loops synchronously on renewal failure. *High.*
- **[P1-G5] Standalone HTTP server has no socket timeouts / connection cap (slowloris)** — `standalone-gateway/server.ts:96-104`. No `requestTimeout`/`headersTimeout`/`keepAliveTimeout`/`maxConnections` (the gateway app set all four in pass 1; the standalone — internet-facing in webhook mode — did not). **Fix:** mirror `apps/gateway/src/daemon.ts:155-158`. *High.* (See also P1-O5: the cloud web role has the same gap.)

### Correctness
- **[P1-X1] Lost-update race on cloud custom-content settings (read-modify-write across `await`, unserialized)** — `workspace-gateway.ts:1741-1758`. `upsertCloudItemSetting`/`removeCloudItemSetting` fetch the entire item array (async), rebuild, and write it all back (async) with no mutex; the custom-agent *edit* flow is two back-to-back RMW cycles. Concurrent saves last-write-win → a saved agent/MCP/skill silently vanishes, no error. **Fix:** serialize per `(workspaceId, keyName)`, or add a server-side atomic append/patch. *High.*
- **[P1-X2] Out-of-order workspace-event delivery to the renderer** — `cloud-workspace-adapter.ts:696-711`, `ipc/workspace-handlers.ts:17-30`. Each event is wrapped in `void (async()=>{ … await sync(); onEvent() })()`, so event N's `await sync()` doesn't gate N+1; the IPC handler then fires N independent `listCloudSessions` fetches whose `.then`s resolve in arbitrary order → `lastEventSequence` can regress and a stale snapshot can overwrite a newer one. (Durable cache cursor is `Math.max`-clamped; only live renderer delivery regresses.) **Fix:** sequence per subscription (`tail = tail.then(...)`); have the renderer drop payloads with `lastEventSequence ≤ last applied`. *High (ordering); med (visible regression).*
- **[P1-X3] Headless-host state file written non-atomically → orphaned runtime** — `headless-host.ts:214`. Uses `fs/promises.writeFile` (not the standard `writeFileAtomic`); a crash mid-write truncates the JSON, and the reader's `catch {}` returns `null` (corrupt reads as "no host running"), orphaning a live `--headless` runtime. **Fix:** `writeFileAtomic`; distinguish parse-error from not-found. *High.*

### Ops / production-readiness
- **[P1-O1] Helm: no `preStop` hook on any pod → SIGTERM races endpoint removal (dropped requests every rollout)** — `helm/open-cowork-cloud/templates/deployment.yaml`, `helm/open-cowork-gateway/...`. `close()` stops accepting connections immediately on SIGTERM while the pod is still a routable endpoint; `maxUnavailable:0/maxSurge:1` doesn't close the kube-proxy propagation race → user-visible 5xx/connection-resets on routine deploys. **Fix:** `lifecycle.preStop: sleep ~10s` on web + gateway with grace > preStop+drain. *High.*
- **[P1-O2] Cloud `close()` doesn't flip `/readyz` to draining and never force-closes hung connections** — `http-server.ts:1439-1448`, `app.ts:1424-1440`. No `draining` flag (so `/readyz` stays 200 during drain → LB keeps routing, compounds O1) and no `server.closeAllConnections()` after a deadline (one hung request blocks shutdown until SIGKILL). **Fix:** set draining→`/readyz` 503 immediately; force-close stragglers after a bounded grace. *High.*
- **[P1-O3] Worker and scheduler pods have NO liveness/readiness probe** — `helm/open-cowork-cloud/templates/deployment.yaml:242-263` (probes gated `if eq role "web"`). A wedged worker/scheduler loop is never restarted; command/workflow processing silently stops with no self-healing. **Fix:** a tiny `/livez` (or heartbeat-file exec probe) reporting last-loop-tick age, wired as `livenessProbe` for those roles. *High.*
- **[P1-O4] The durable-state layer emits ZERO metrics; `object_store_errors_total` is a phantom alert** — `object-store.ts` (no observability import), `workspace-checkpoint-store.ts`. The metric is cataloged, paged on, graphed, and an SLO — but has **0 emission sites**. Object-store get/put/head and checkpoint save/restore are completely dark, so the one alert covering session-state loss can never fire. **Fix:** emit success/error counters + latency histograms via `recordCloudMetric`. *High.* (Related P3: `/metrics` isn't exposed and no ServiceMonitor ships — the whole Prometheus pack depends on undocumented OTLP-collector wiring; `observability.ts` OTLP export also skips the high-cardinality strip list.)
- **[P1-O5] Cloud HTTP server sets no socket timeouts / `maxConnections`** — `http-server.ts:1417`. The public web role (browser UI + pre-auth `/api/invites/accept`, `/webhooks/billing`, `/webhooks/workflows/*`) has no slowloris/connection-exhaustion protection (Node's 300s `requestTimeout` is weak; no connection ceiling). **Fix:** apply the gateway's four timeouts + `maxConnections`. *High.* (Same root gap as P1-G5.)
- **[P1-O6] In-memory↔Postgres parity contract omits the claim/fencing methods; `listRunnableSessions` has no test anywhere** *(found by 2 auditors)* — `tests/cloud-control-plane-store-contracts.test.ts`. `runControlPlaneDomainContracts` never exercises `claimChannelProviderEvent`, `claimNextChannelDelivery`, `claimNextSessionCommand`, or `listRunnableSessions`; the concurrency proofs (`cloud-postgres-concurrency.test.ts`) are all `skip: POSTGRES_SKIP` so a default `pnpm test` runs none of them. These are exactly the lease-fencing/claim-atomicity methods where a subtle SQL difference causes prod double-claims but passes in-memory. **Fix:** add those four to the always-on (in-memory + pglite) contract; port the concurrency proofs to pglite. *High.*

## P2 — Medium

### Security / crypto
- **[P2-1] `enc:v1` envelope has no key id → no safe key rotation** — `secret-adapter.ts:24-29,409-445`. The envelope carries no `kid`; rotating `OPEN_COWORK_CLOUD_SECRET_KEY` makes all stored BYOK ciphertexts undecryptable with no migration path. **Fix:** add `kid` + a keyring (current+previous) so reveal selects by kid, protect uses newest.
- **[P2-2] Markdown sanitizer doesn't force `rel="noopener noreferrer"` on `target="_blank"` and allows arbitrary `id` (DOM-clobbering)** — `MarkdownContent.tsx:17-22`. The widest untrusted-content surface (LLM/channel markdown) relies on Chromium's implicit `noopener` default. **Fix:** `afterSanitizeAttributes` hook forcing `rel` + a protocol allowlist.
- **[P2-3] Mermaid sanitize profile permits `<foreignObject>` and the code then trusts post-sanitize foreignObject styling** — `MermaidChart.tsx:42-47,107-121`. With `htmlLabels:false` it shouldn't appear; defense-in-depth. **Fix:** `FORBID_TAGS:['foreignObject']` or document why svg-only is sufficient.
- **[P2-4] Weak SSRF classifier in capability-bundle engine (regex host parse; `169.254.169.254` classified *public*)** — `capability-bundle-engine.ts:307-315`. Diverges from `mcp-url-policy`; only a compatibility flag today (never fetches the URL), so not a live SSRF — but a footgun if reused as authorization. **Fix:** delete it; call `evaluateHttpMcpUrl`.
- **[P2-5] `provider-catalog` outbound fetch enforces only `https:` (no private-IP block) and attaches an auth header** — `provider-catalog.ts:160-167,214`. URL source is the local trusted config today; becomes a metadata-exfil SSRF if descriptors ever come from a less-trusted channel. **Fix:** route through `evaluateHttpMcpUrlResolved`.
- **[P2-6] Hardcoded constant salt (no pepper) for worker-credential / channel-interaction / legacy-token hashes** — `in-memory-domains/workers.ts:398-400`, `control-plane-tokens.ts:27-29,45-47`. `scryptSync(plaintext, '<constant>', 32)` vs the correct per-token random-salt v2 API-token scheme. Mitigated by 256-bit entropy; a DB-only compromise permits precomputation. **Fix:** per-credential random salt (+ optional KMS pepper).

### Cloud scalability
- **[P2-7] Concurrency-counter triggers serialize the hottest writes on one counter row and can drift** — `postgres-schema.ts:920-1023`. All active-state writes for an org contend on a single `cloud_concurrency_counters` row; each firing also runs a per-row `SELECT org_id FROM cloud_orgs`; the `GREATEST(0, value+delta)` clamp on the in-place UPDATE permanently loses any momentary-negative decrement with **no reconciliation** outside the one-shot backfill. **Fix:** clamp only on read (not UPDATE); add a periodic reconcile job; consider sharding the row.
- **[P2-8] Unbounded per-org list reads** (no LIMIT/keyset): `listChannelBindings` `channel-bindings.ts:162`, `listApiTokens` `api-tokens.ts:123`, `listByokSecrets` `byok.ts:140` (+ forced sort), `listThreadSmartFilters` `thread-index.ts:153` (PK-only, full sort), and worst — `listWorkerHeartbeats` `worker-heartbeats.ts:50` (`SELECT * … ORDER BY worker_id`, **no WHERE, no LIMIT** = cross-tenant full scan, called from workspace routes). **Fix:** LIMIT/keyset + covering indexes; bound `listWorkerHeartbeats`.
- **[P2-9] No server-level socket/idle timeout for the cloud SSE sockets; half-open peers detected only when the OS buffer fills** — `http-server.ts:1417`, `channel-delivery-sse.ts:96`. Compounds P0-1/P1-C1. **Fix:** `socket.setKeepAlive(true,…)` on SSE sockets + a max SSE lifetime.
- **[P2-10] Gateway channel-delivery SSE has no slow-client backpressure cap** — `http-routes/channel-delivery-sse.ts:31-35`. Unlike session/workspace SSE (which drop past `SSE_MAX_BUFFERED_BYTES`), this writer never checks `res.writableLength` → per-connection heap growth → OOM. **Fix:** apply the same 8 MiB drop cap.
- **[P2-11] Worker-scoped runtime cache evicts only on cache-miss — idle OpenCode child processes never reaped on a timer** — `worker-scoped-runtime-adapter.ts:157-180`. Up to `maxRuntimeEntries`(100) child processes + workspaces linger after sessions go quiet. **Fix:** periodic `evictRuntimes()` timer (~`idleTtl/2`).

### Correctness / reliability
- **[P2-12] Encrypted credential/cache files deleted on a *transient* decrypt failure** — `desktop-pairing/credentials.ts:179-185`, `cloud-workspace-credentials.ts:193-198`, `cloud-workspace-cache.ts:507-512`. A momentary `safeStorage` hiccup (keychain locked) → `rmSync` destroys pairing tokens / cloud OAuth tokens, forcing full re-pair/re-login. **Fix:** rename to `*.corrupt` for diagnosis; require a post-decrypt JSON-parse failure (non-transient) before deleting.
- **[P2-13] `readState`/`readRecords` swallow parse errors → "empty", then the next write clobbers good-but-unreadable data** — `desktop-pairing/store.ts:342-345`, `cloud-workspace-registry.ts:183`, `gateway-workspace-registry.ts:171`. A corrupt/half-written file is indistinguishable from "no entries"; the next `save` overwrites with only the new entry → total prior-data loss. **Fix:** distinguish absent from unparseable; refuse the write (or back up) on parse failure.
- **[P2-14] Floating promises / missing `.catch` that can crash or hang** — `provider-catalog.ts:297` (`.then` with no `.catch`; unguarded `mkdirSync` → unhandled rejection → crash), `auth.ts:461` (`void openExternal(...)` rejection escapes the sync try/catch → login hangs to the 120s timeout), `index.ts:442` (the large `app.whenReady().then(async …)` startup body has no `.catch`, so a recoverable startup hiccup escalates to full process exit). **Fix:** add targeted `.catch`/try-catch with specific logging.
- **[P2-15] `handleMessage` claims a provider event before the prompt completes; a post-send timeout can duplicate or silently drop the message** — `gateway-runtime.ts:165-257`. Non-cloud errors mark non-retryable → silent drop with no channel feedback; a timeout after the cloud accepted → retry duplicates. Load-bearing assumption: cloud dedups on `commandId==eventId`. **Fix:** confirm cloud idempotency on that key; classify post-send timeouts conservatively; surface a channel-visible failure on permanent drop.
- **[P2-16] `classifyProviderFailure` defaults *unknown* errors to retryable; non-webhook sends carry no idempotency key** — `provider-errors.ts:41-44`, `gateway-runtime.ts:312-330`. Message-regex classification is brittle and a transient-looking error after a partial multi-chunk send can duplicate user-visible messages. **Fix:** default unknown → non-retryable (allowlist transient codes); require an idempotency key before retrying any non-webhook send.

### Ops
- **[P2-17] Production-safety asserts only fire at `tier==='public_production'`; the default `local` tier accepts weak/missing secrets, and the compose files bake publicly-known secrets + `ALLOW_INSECURE_AUTH=true`** — `app.ts:734,241-243`, `docker-compose.cloud.yml:20,31-33`. ## P3 — Low / hardening

- **[P3-1] CSRF token compared with non-constant-time `!==`** *(2 auditors)* — `session-cookie-auth.ts:203`. Negligible (the token lives inside the HMAC-signed cookie the user already holds) but inconsistent with the file's constant-time discipline. **Fix:** `constantTimeEquals`.

- **[P3-10] Standalone DB JSON `JSON.parse` not wrapped in try/catch** — `standalone-gateway/postgres-repository.ts:682`. A corrupt `metadata` column throws out of the row mapper (cloud-server readers catch). **Fix:** wrap → `{}`.
