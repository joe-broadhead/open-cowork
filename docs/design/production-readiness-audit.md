# Open Cowork ‚Äî Production-Readiness Audit (whole product)

**Date:** 2026-06-16 ¬∑ **Scope:** product, features, design-vs-brief, security, scalability, modularity/code-quality, downstream configurability, deployment topologies. **Method:** seven parallel specialist audits, each grounded in `file:line` evidence verified by reading. This document consolidates and prioritizes them.

> This is the broad *product* audit. The narrower Studio design/parity remediation log lives in [`studio-production-audit.md`](./studio-production-audit.md).

---

## 1. Verdict

**Open Cowork is a genuinely substantial, well-architected platform ‚Äî not a demo shell ‚Äî but it is NOT yet production-ready for a public multi-tenant launch.** The engineering substrate is strong: parameterized + tenant-scoped SQL (no SQLi/IDOR in the Postgres control plane), HMAC-verified + replay-protected webhooks, AES-256-GCM BYOK envelopes, correct OIDC/JWT validation, per-request CSP nonces, a clean `all-in-one | web | worker | scheduler` role model with real Postgres-backed HA (SKIP LOCKED leasing + reaper), and high code discipline (0 `as any`, 0 `@ts-ignore`, 0 swallowed catches in prod, enforced coverage, custom lint gates).

Launch is gated on **three P0 clusters**: (1) **scalability ceilings** that will fail under public load, (2) **two flagship features that look done but are broken end-to-end** (BYOK activation, team invites), and (3) the **Knowledge wiki being half-productized and architecturally off-platform**. Two cross-cutting goals the owner emphasized ‚Äî *downstream configurability* and *public-app polish* ‚Äî are each ~75% there with specific, enumerated gaps.

### Readiness scorecard

| Dimension | Grade | One-line |
|---|---|---|
| Security | **B+** | Strong fundamentals; a few fail-open-on-misconfig hazards + knowledge tenant-isolation regression |
| Scalability / performance | **C ‚Äî not ready** | Critical ceilings: unconfigured PG pool, N+1 claim, gateway FD/heap leak, per-connection SSE DB polling |
| Feature completeness | **C+** | Substantial + real, but 2 "broken-looking-done" features + cloud/desktop parity gaps |
| Downstream configurability | **B‚àí** | Mature config surface; desktop has no feature flags, dead `entitlements.features`, white-label icon/favicon gaps |
| Deployment topologies | **B+** | Excellent role model; the "cloud server lives inside apps/desktop" coupling is the one structural wart |
| Code quality / modularity | **A‚àí** | High discipline; toothless line budgets, 2 remaining monoliths, 1 untested prod persistence layer |
| Design / parity / polish | **B** | Strong shared CSS substrate; web empty/loading states unstyled, web missing Permissions/Models |

## 1a. Remediation log (in progress ‚Äî goal: fix all findings end to end)

**P0-1 scale hardening ‚Äî 5/‚Äçmany fixed (verified, full gate green at each step):**
- ‚úÖ **Postgres pool unconfigured** ‚Üí added `postgres-pool-options.ts` (env-driven `max` + connection/idle timeouts + `statement_timeout`/`idle_in_transaction_session_timeout`/`lock_timeout`); safe defaults preserve behaviour (statement/lock opt-in so migrations aren't truncated) + a 2-min idle-in-transaction guard bounds leaked-lock holds. Unit-tested (`cloud-postgres-pool-options.test.ts`, 4/4).
- ‚úÖ **`findApiTokenByPlaintext` cross-org full scan** (auth hot path) ‚Üí O(1) primary-key resolution via an anchored token-id parse, with a fallback scan for non-standard ids; hash still verified (new contract assertion: tampered-secret on a valid id fails).
- ‚úÖ **`claimRunnableSessions` N+1** ‚Üí batch-fetch all selected sessions' leases in one query instead of a per-row locking `getLease` (the per-session `FOR UPDATE ‚Ä¶ SKIP LOCKED` already serializes claims, so the per-row lock was redundant).
- ‚úÖ **Gateway stream/FD leak** ‚Üí `session-stream-manager` is now a bounded cache: idle-TTL + hard-cap LRU eviction (lazy on `ensure` + periodic unref'd sweep), `streamEvictions` metric + Prometheus counter; evicted idle streams re-subscribe from the persisted cursor on the next message. 2 new gateway tests (cap + TTL); gateway suite 81/81.
- ‚úÖ **`listThreadMetadata` N+1** ‚Üí one batched `session_id = ANY($2)` tag fetch grouped in memory (was up to 500 per-session queries).

- ‚úÖ **SSE replay polled an unbounded `SELECT *` per topic** ‚Üí `listSessionEvents` now takes an optional `limit` (default unbounded, so projection rebuild is unaffected); the SSE replay path reads in bounded `SSE_REPLAY_BATCH` (1,000) pages, and the replay hub re-polls immediately on a full batch so a backlog drains without per-page `pollMs` waits ‚Äî delivery stays complete (cursor advances per batch) and the cross-replica polling model is preserved. Contract asserts the bound on both stores; cloud-http-server 100/100, sse-replay 2/2.

- ‚úÖ **SSE workspace-event read bound** ‚Üí `listWorkspaceEvents` got the same optional `limit` + bounded SSE replay + batch re-poll as session events (symmetry; default unbounded for non-SSE callers).
- ‚úÖ **SSE write backpressure** ‚Üí each SSE connection now caps Node's outbound writable buffer (`res.writableLength`) at 8 MB and drops the connection past it (cleanup unsubscribes on close), so a slow client can't balloon server heap. Both SSE handlers; cloud-http-server 100/100.

The SSE polling ceiling is now bounded end-to-end (both event reads paginated + write backpressure capped + cross-replica polling preserved); the optional LISTEN/NOTIFY push remains a future optimization over the now-bounded polling.
- ‚úÖ **Unbounded `listWorkflows` / `listAllSessions`** ‚Üí defensive caps (per-user workflows 500, both stores; cross-tenant `listAllSessions` scan 1000, no prod caller). Keyset pagination remains the future enhancement; the bound removes the unbounded-scan risk. Contract 3/3, full suite 2,024/0.

**P0-1 critical scalability ceilings are now closed** (pool, token-scan, claim N+1, gateway FD-leak, thread-metadata N+1, SSE reads+backpressure, unbounded lists).

- ‚úÖ **Per-write quota billing round-trips halved** ‚Üí `commandQueueQuotaForOrg` (every prompt enqueue) and `workflowRunQuotaForOrg` (every workflow run) each called `effectiveQuotaLimit` twice, doing **two** `getBillingSubscription` SELECTs per write to resolve two entitlement limits. Refactored to `resolveOrgEntitlements()` (one fetch) + `limitFromEntitlements()` (pure read) so each path now does **one** subscription round-trip ‚Äî a pure refactor with identical limits and no cross-request staleness. New focused test (`cloud-usage-governance` 3/3) asserts the single-fetch behaviour (and zero fetch when billing is off); quota behaviour unchanged (cloud-http-server 104/104). Full node 2,035/0.

**Remaining P0-1 Medium perf ‚Äî assessed, with rationale for what's deferred:**
- ‚óê **Concurrency-quota `COUNT(*)` aggregates** (queued-commands per enqueue, active-workers per lease claim, concurrent-sessions, workflow-runs): each is bounded by the quota limit itself (you can't exceed `maxQueued‚Ä¶` pending rows), so the scan is over a small live set; the durable fix is maintained gauge counters (mirroring the existing windowed `cloud_usage_counters`), but that adds drift/reconciliation risk on a correctness-sensitive path ‚Äî deferred as a deliberate, separately-verified change rather than rushed. The windowed/rate quotas already use O(1) counters.
- ‚óê **Auth re-validation round-trips:** the clean fix (a short-TTL validated-token/principal cache) would delay API-token **revocation** by up to the TTL ‚Äî which the project's security constraints explicitly forbid ("do not weaken API-token verification"). The alternative (an `ensurePrincipal` fast-path) can't safely skip the bootstrap upserts because `ensureOrgForTenant` performs the **org-active security gate** and the upserts feed role resolution. So this is intentionally left as-is: correctness/security is preferred over the round-trip saving until a revocation-safe design (e.g. a revocation-list-aware cache) is built. Cookie + OIDC auth are already stateless.

## 2. P0 ‚Äî Production blockers (fix before any public launch)

### P0-1 ¬∑ Scalability ceilings that fail under public load
These are *ceilings*, not correctness bugs (concurrency correctness is proven by `cloud-postgres-concurrency.test.ts`), but each will take down a public deployment.

- **Postgres pool is unconfigured** ‚Äî `postgres-control-plane-store.ts:181` (`new pg.Pool({ connectionString })`): defaults to **max 10 connections/process**, and there is **no `statement_timeout` / `lock_timeout` / `idle_in_transaction_session_timeout`** anywhere. Every mutation holds a connection for a full `BEGIN‚ÜíCOMMIT`; a few slow/blocked txns exhaust the pool and stall the whole control plane. ‚Üí Make pool size + all three timeouts env-configurable; size against DB `max_connections √∑ replicas`.
- **`claimWork` is an N+1 inside the claim transaction** ‚Äî `postgres-control-plane-store.ts:1126-1198`: selects up to 1000 rows `FOR UPDATE ‚Ä¶ SKIP LOCKED`, then loops per row doing `getLease(forUpdate)` + `UPDATE` + lease upsert ‚Äî up to ~3000 serial queries holding locks. ‚Üí Collapse to one set-based `UPDATE ‚Ä¶ FROM (cte) RETURNING`.
- **Gateway stream/render state + upstream SSE sockets are never evicted** ‚Äî `apps/gateway/src/session-stream-manager.ts:54` (`streams` Map, no terminal-event close; `close()` has zero prod callers) + `apps/gateway/src/render/state.ts:20`: grows O(all sessions ever seen) in **heap and file descriptors** ‚Äî a public deployment exhausts FDs within hours. ‚Üí Evict on terminal events + idle-TTL/LRU + hard `maxStreams` cap.
- **SSE is per-connection DB polling of an unbounded `SELECT *`** ‚Äî `http-server.ts:581-591` polls `listSessionEvents` (`postgres-control-plane-store.ts:1302`, no `LIMIT`) every ~1s per connection; an `afterSequence=0` client re-pulls full history each poll. N clients ‚áí ~N unbounded scans/sec on the 10-slot pool. ‚Üí `LIMIT`+cursor on event reads; drive fan-out from the existing in-process `eventBus` / Postgres `LISTEN/NOTIFY`, not per-client polling. Also: **SSE writes ignore backpressure** (`http-sse-helpers.ts:28`, `res.write` return unused) ‚Üí slow clients balloon server heap.
- **Auth hot path is a full table scan** ‚Äî `findApiTokenByPlaintext` (`postgres-store-domains/api-tokens.ts:85`) computes its match key per-row (`left($1, length('occ_'||token_id||'_'))`), non-sargable, scanning **all live tokens across all orgs** on every token-authed request. ‚Üí Encode a lookup-able token-id prefix and resolve via the existing `UNIQUE(token_hash)` in O(1) (the worker-credential path already does this).
- Plus several **unbounded list queries** (`listAllSessions` is a cross-tenant full scan with an unindexed global `ORDER BY`; `listWorkflows`, `listSessionEvents`, `listApiTokens`) and `listThreadMetadata`'s **N+1 tag fetch** (up to 501 queries/call, `thread-index.ts:255`).

## 4. Deployment topologies (separate or combined)

**Assessment: B+ ‚Äî among the better examples of this pattern.** A downstream builder can deploy **gateway-only** (two well-defined shapes: standalone w/ own Postgres, or cloud-channel appliance), **cloud-only** (split `web`/`worker`/`scheduler`), or **combined** (`all-in-one`) ‚Äî switching shape on one env var (`OPEN_COWORK_CLOUD_ROLE`) with no rebuild. Wiring is fully env/secret-driven (anti-hardcoding is a documented, enforced rule), the prod cloud image is Electron-free, non-root, digest-pinned, role-aware, and shared packages are leak-free. Seven topology profiles are captured in a machine-readable contract (`deploy/topologies/topology-profiles.json`).

The one structural wart + two ergonomics gaps:
- **The cloud server source lives inside `apps/desktop` and its runtime graph value-imports Electron.** (Investigated 2026-06: the cloud entry actually *does* value-import `electron` via `config-loader.ts`/`settings.ts`/etc. ‚Äî Electron-freedom is **not** import-graph purity.) The real mechanism is `scripts/build-cloud.mjs`'s `cloudElectronShimPlugin`: a **hardcoded list of 15 electron exports** (`app`, `safeStorage`, `net`, `protocol`, `shell`, `session`, `BrowserWindow`, `ipcMain`, `Menu`, `nativeImage`, `dialog`, `Notification`, `Tray`, `powerMonitor`, `utilityProcess`) all bound to `undefined`, plus the cloud code's defensive `(electron as { app? }).app` nullable guards. So a "no `electron` in the cloud graph" test is the *wrong* invariant pre-extraction (it fails on the legitimate shimmed value-imports). The **precise residual risk**: a future cloud-graph file that imports/uses an electron API **not in those 15** (e.g. `clipboard`, `globalShortcut`, `screen`) compiles, ships, and crashes at runtime ‚Äî the shim silently returns `undefined`. ‚Üí Two viable guards: (a) extract the cloud server into its own workspace package (`packages/cloud-server` / `apps/cloud`) so the dependency is structural; or (b) a test asserting every electron name used by the **value-reachable** cloud graph is covered by the shim's export list (needs value-vs-type-aware import-graph following ‚Äî `import type` targets are erased and must not be followed). Either makes the shim's completeness enforceable instead of manual.
- ‚úÖ **Ephemeral storage no longer silent on beta tiers.** `startCloudApp` now calls `describeUnacknowledgedEphemeralStorage` after resolving the store + object store: for `self_host_beta`/`private_beta` (the tiers the production asserts explicitly permit ephemeral on ‚Äî "filesystem ‚Ä¶ are local/self-host-beta only"), an in-memory control plane or filesystem/unavailable object store emits a loud `recordCloudLog({level:'warn', name:'cloud.storage.ephemeral'})` naming exactly which backend loses state on restart, unless the operator sets `OPEN_COWORK_CLOUD_ALLOW_EPHEMERAL_STORAGE=true` to acknowledge. `public_production` keeps its existing unconditional hard-throws (the opt-in cannot weaken it); `local` stays quiet. ## 6. Consolidated findings by dimension (severity-ordered, deduped)

> P0 items above are not repeated here. This is the High/Medium backlog.

### Security (beyond P0 knowledge isolation)
- **High ‚Äî OIDC self-service "open" mode auto-joins on a self-asserted `tenant_id` claim** (`oidc-auth.ts:252` ‚Üí `session-service.ts:656-692`). Safe by default (production forces `invite`), but a dangerous-config hazard enabling cross-tenant access against a multi-tenant IdP. ‚Üí Bind `tenant_id` to an allowlisted issuer; never auto-join a *pre-existing* org from a claim; refuse `open` mode without `allowedEmailDomains`.
- **Medium ‚Äî Rate-limiting + auth-backoff fail open by config and on store error** ‚Äî `session-service.ts:2509/2525/2534` no-op when `abuse.enabled` is false; `http-server.ts:1469` proceeds when the backoff store throws. ‚Üí Default abuse-protection ON for the public profile (or gate readiness on it); fail-closed on backoff-store errors for auth.
- **Medium ‚Äî Plaintext (base64) secret adapter** is opt-in-strong-key-only and allowed below `public_production` (`secret-adapter.ts:447`). ‚Üí Require strong key / reject plaintext at startup for any BYOK-persisting tier.
- **Low** ‚Äî caller-discipline (not in-query) tenant scoping in `requireCommand`/`revokeApiToken`/`getProposal`; standalone-gateway allows `0.0.0.0` bind with no opt-in; log-redaction misses unprefixed Slack/Telegram secrets; branding "complex" CSS tokens skip color-parse (operator-only surface).

**Confirmed solid:** SQLi (none ‚Äî uniformly parameterized), Postgres tenant isolation, API-token hashing (scrypt + constant-time + fail-closed), BYOK AES-256-GCM w/ per-secret AAD + never-logged plaintext, OIDC RS256-only + full claim checks + PKCE, webhook HMAC + replay + fail-closed across all providers, CSP nonce + strict CORS + HttpOnly/Secure/SameSite cookies + CSRF, static-asset path-traversal safety, supply-chain hygiene (locked, pinned overrides, security eslint plugins).
