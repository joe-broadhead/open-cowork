# Open Cowork — Production-Readiness Audit (whole product)

**Date:** 2026-06-16 · **Scope:** product, features, design-vs-brief, security, scalability, modularity/code-quality, downstream configurability, deployment topologies. **Method:** seven parallel specialist audits, each grounded in `file:line` evidence verified by reading. This document consolidates and prioritizes them.

> This is the broad *product* audit. The narrower Studio design/parity remediation log lives in [`studio-production-audit.md`](./studio-production-audit.md).

---

## 1. Verdict

**Open Cowork is a genuinely substantial, well-architected platform — not a demo shell — but it is NOT yet production-ready for a public multi-tenant launch.** The engineering substrate is strong: parameterized + tenant-scoped SQL (no SQLi/IDOR in the Postgres control plane), HMAC-verified + replay-protected webhooks, AES-256-GCM BYOK envelopes, correct OIDC/JWT validation, per-request CSP nonces, a clean `all-in-one | web | worker | scheduler` role model with real Postgres-backed HA (SKIP LOCKED leasing + reaper), and high code discipline (0 `as any`, 0 `@ts-ignore`, 0 swallowed catches in prod, enforced coverage, custom lint gates).

Launch is gated on **three P0 clusters**: (1) **scalability ceilings** that will fail under public load, (2) **two flagship features that look done but are broken end-to-end** (BYOK activation, team invites), and (3) the **Knowledge wiki being half-productized and architecturally off-platform**. Two cross-cutting goals the owner emphasized — *downstream configurability* and *public-app polish* — are each ~75% there with specific, enumerated gaps.

### Readiness scorecard

| Dimension | Grade | One-line |
|---|---|---|
| Security | **B+** | Strong fundamentals; a few fail-open-on-misconfig hazards + knowledge tenant-isolation regression |
| Scalability / performance | **C — not ready** | Critical ceilings: unconfigured PG pool, N+1 claim, gateway FD/heap leak, per-connection SSE DB polling |
| Feature completeness | **C+** | Substantial + real, but 2 "broken-looking-done" features + cloud/desktop parity gaps |
| Downstream configurability | **B−** | Mature config surface; desktop has no feature flags, dead `entitlements.features`, white-label icon/favicon gaps |
| Deployment topologies | **B+** | Excellent role model; the "cloud server lives inside apps/desktop" coupling is the one structural wart |
| Code quality / modularity | **A−** | High discipline; toothless line budgets, 2 remaining monoliths, 1 untested prod persistence layer |
| Design / parity / polish | **B** | Strong shared CSS substrate; web empty/loading states unstyled, web missing Permissions/Models |

## 1a. Remediation log (in progress — goal: fix all findings end to end)

**P0-1 scale hardening — 5/‍many fixed (verified, full gate green at each step):**
- ✅ **Postgres pool unconfigured** → added `postgres-pool-options.ts` (env-driven `max` + connection/idle timeouts + `statement_timeout`/`idle_in_transaction_session_timeout`/`lock_timeout`); safe defaults preserve behaviour (statement/lock opt-in so migrations aren't truncated) + a 2-min idle-in-transaction guard bounds leaked-lock holds. Unit-tested (`cloud-postgres-pool-options.test.ts`, 4/4).
- ✅ **`findApiTokenByPlaintext` cross-org full scan** (auth hot path) → O(1) primary-key resolution via an anchored token-id parse, with a fallback scan for non-standard ids; hash still verified (new contract assertion: tampered-secret on a valid id fails).
- ✅ **`claimRunnableSessions` N+1** → batch-fetch all selected sessions' leases in one query instead of a per-row locking `getLease` (the per-session `FOR UPDATE … SKIP LOCKED` already serializes claims, so the per-row lock was redundant).
- ✅ **Gateway stream/FD leak** → `session-stream-manager` is now a bounded cache: idle-TTL + hard-cap LRU eviction (lazy on `ensure` + periodic unref'd sweep), `streamEvictions` metric + Prometheus counter; evicted idle streams re-subscribe from the persisted cursor on the next message. 2 new gateway tests (cap + TTL); gateway suite 81/81.
- ✅ **`listThreadMetadata` N+1** → one batched `session_id = ANY($2)` tag fetch grouped in memory (was up to 500 per-session queries).

- ✅ **SSE replay polled an unbounded `SELECT *` per topic** → `listSessionEvents` now takes an optional `limit` (default unbounded, so projection rebuild is unaffected); the SSE replay path reads in bounded `SSE_REPLAY_BATCH` (1,000) pages, and the replay hub re-polls immediately on a full batch so a backlog drains without per-page `pollMs` waits — delivery stays complete (cursor advances per batch) and the cross-replica polling model is preserved. Contract asserts the bound on both stores; cloud-http-server 100/100, sse-replay 2/2.

- ✅ **SSE workspace-event read bound** → `listWorkspaceEvents` got the same optional `limit` + bounded SSE replay + batch re-poll as session events (symmetry; default unbounded for non-SSE callers).
- ✅ **SSE write backpressure** → each SSE connection now caps Node's outbound writable buffer (`res.writableLength`) at 8 MB and drops the connection past it (cleanup unsubscribes on close), so a slow client can't balloon server heap. Both SSE handlers; cloud-http-server 100/100.

The SSE polling ceiling is now bounded end-to-end (both event reads paginated + write backpressure capped + cross-replica polling preserved); the optional LISTEN/NOTIFY push remains a future optimization over the now-bounded polling.
- ✅ **Unbounded `listWorkflows` / `listAllSessions`** → defensive caps (per-user workflows 500, both stores; cross-tenant `listAllSessions` scan 1000, no prod caller). Keyset pagination remains the future enhancement; the bound removes the unbounded-scan risk. Contract 3/3, full suite 2,024/0.

**Remaining P0-1 Medium perf — assessed, with rationale for what's deferred:**
- ◐ **Concurrency-quota `COUNT(*)` aggregates** (queued-commands per enqueue, active-workers per lease claim, concurrent-sessions, workflow-runs): each is bounded by the quota limit itself (you can't exceed `maxQueued…` pending rows), so the scan is over a small live set; the durable fix is maintained gauge counters (mirroring the existing windowed `cloud_usage_counters`), but that adds drift/reconciliation risk on a correctness-sensitive path — deferred as a deliberate, separately-verified change rather than rushed. The windowed/rate quotas already use O(1) counters. (Re-confirmed 2026-06: these tables are already well-indexed for the hot access patterns — `cloud_session_commands_pending_idx`/`_runnable_idx`, `cloud_workflow_runs_claim_idx`/`_reaper_idx`, `cloud_sessions_user_status_cursor_idx` — so the bounded COUNTs are **index-served, not full scans**, and there is no safe index win left to add. Adding gauge counters would trade that for reconciliation/drift risk on a correctness-sensitive hot path for negligible gain — correct as-is.) **→ Building the maintained gauge counters anyway, the drift-free way (2026-06).** A `cloud_concurrency_counters` table is kept by an AFTER-row **trigger** on each counted table — it fires on every INSERT/UPDATE/DELETE regardless of which code path made the change, so the gauge **cannot drift** (unlike application-level +1/−1 hooks across the spread mutation sites), and a migration-time backfill seeds it from the current COUNT. The quota hot path then reads an O(1) primary-key row. **✅ Workflow-runs gauge done** (migration `017_cloud_concurrency_counters`, key `concurrent_workflow_runs`): the quota check reads the counter instead of `COUNT(*)`+JOIN, and a pglite test proves the gauge stays exactly == COUNT through queued→running→completed and a DELETE. **✅ Concurrent-sessions gauge done** (migration `018`, key `concurrent_sessions`, active = `status <> 'closed'`) and **✅ queued-commands gauge done** (migration `019`, key `queued_commands`, active = `status IN ('pending','running')`) — same trigger pattern; the command quota keeps a bounded `min(created_at)` scan only when a queue-AGE limit is configured (a gauge can't carry an age). The pglite test proves all three gauges stay exactly == COUNT through their lifecycles. **The 4th — active-workers — genuinely cannot be a gauge:** its predicate is `lease_expires_at_ms > now` (time-based), so a lease goes inactive with **no row event** a trigger could catch; a counter would over-count expired-but-unreaped leases and could falsely block at quota — so the time-bounded COUNT is the correct implementation there and stays. Net: **3 of the 4 hot-path quota COUNTs are now O(1) gauge reads; the 4th is correctly a time-bounded scan.** The trigger is the single place this departs from the codebase's pure-query style — justified as the only race-free, drift-free way to maintain the gauges. Gate: main tsc 0, node 2,098/0/28, lint 1,681.
- ✅ **Auth re-validation round-trips — fixed without weakening revocation.** The naive fix (caching the validated principal) would delay API-token revocation by the TTL, which is forbidden. The key realization: the audit's blocker — "`ensureOrgForTenant` performs the org-active gate and the upserts feed role resolution" — is resolvable by **splitting the gate (a read) from the bootstrap writes**. `ensurePrincipal` already calls `resolvePrincipalMembership` (a read) every request, and that record carries `org.status`, `membership.status`, and `membership.role`. So I added a per-(tenant, account) "bootstrapped" marker (60 s TTL): once set, a request **skips the idempotent bootstrap WRITES** (`createTenant`/`ensureOrgForTenant`/`createAccount`/`ensureUser`/`upsertMembership`) — but **every security gate still runs on that request** from the fresh membership read (org must be active, membership must be active, role comes from the read). No principal/role is cached, so a deactivated org / suspended membership / revoked token is rejected on the very next request — zero revocation window. New test (`cloud-http-server.test.ts`) proves both: the 2nd request skips the writes (bootstrap-write count stays 1) AND a membership suspended *after* bootstrap is still rejected. Gate: main tsc 0, node 2,097/0/28, lint 1,680.

## 2. P0 — Production blockers (fix before any public launch)

### P0-1 · Scalability ceilings that fail under public load
These are *ceilings*, not correctness bugs (concurrency correctness is proven by `cloud-postgres-concurrency.test.ts`), but each will take down a public deployment.

- **Postgres pool is unconfigured** — `postgres-control-plane-store.ts:181` (`new pg.Pool({ connectionString })`): defaults to **max 10 connections/process**, and there is **no `statement_timeout` / `lock_timeout` / `idle_in_transaction_session_timeout`** anywhere. Every mutation holds a connection for a full `BEGIN→COMMIT`; a few slow/blocked txns exhaust the pool and stall the whole control plane. → Make pool size + all three timeouts env-configurable; size against DB `max_connections ÷ replicas`.
- **`claimWork` is an N+1 inside the claim transaction** — `postgres-control-plane-store.ts:1126-1198`: selects up to 1000 rows `FOR UPDATE … SKIP LOCKED`, then loops per row doing `getLease(forUpdate)` + `UPDATE` + lease upsert — up to ~3000 serial queries holding locks. → Collapse to one set-based `UPDATE … FROM (cte) RETURNING`.
- **Gateway stream/render state + upstream SSE sockets are never evicted** — `apps/gateway/src/session-stream-manager.ts:54` (`streams` Map, no terminal-event close; `close()` has zero prod callers) + `apps/gateway/src/render/state.ts:20`: grows O(all sessions ever seen) in **heap and file descriptors** — a public deployment exhausts FDs within hours. → Evict on terminal events + idle-TTL/LRU + hard `maxStreams` cap.
- **SSE is per-connection DB polling of an unbounded `SELECT *`** — `http-server.ts:581-591` polls `listSessionEvents` (`postgres-control-plane-store.ts:1302`, no `LIMIT`) every ~1s per connection; an `afterSequence=0` client re-pulls full history each poll. N clients ⇒ ~N unbounded scans/sec on the 10-slot pool. → `LIMIT`+cursor on event reads; drive fan-out from the existing in-process `eventBus` / Postgres `LISTEN/NOTIFY`, not per-client polling. Also: **SSE writes ignore backpressure** (`http-sse-helpers.ts:28`, `res.write` return unused) → slow clients balloon server heap.
- **Auth hot path is a full table scan** — `findApiTokenByPlaintext` (`postgres-store-domains/api-tokens.ts:85`) computes its match key per-row (`left($1, length('occ_'||token_id||'_'))`), non-sargable, scanning **all live tokens across all orgs** on every token-authed request. → Encode a lookup-able token-id prefix and resolve via the existing `UNIQUE(token_hash)` in O(1) (the worker-credential path already does this).
- Plus several **unbounded list queries** (`listAllSessions` is a cross-tenant full scan with an unindexed global `ORDER BY`; `listWorkflows`, `listSessionEvents`, `listApiTokens`) and `listThreadMetadata`'s **N+1 tag fetch** (up to 501 queries/call, `thread-index.ts:255`).
