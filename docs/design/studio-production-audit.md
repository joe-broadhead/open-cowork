## 0. Executive verdict

The **engineering substrate is already world-class**: security is hardened with an explicit threat model, the quality gates (custom lint + ESLint + typecheck + 369 test files) are real and green, error handling and type-safety metrics are excellent. **Do not rewrite these.**

The problem is the **design + parity layer**:

1. **Design fidelity — failing.** Of the brief's 6 named fixes, **none is fully satisfied in both apps.** The app still ships the "flashy AI dashboard" decoration the brief explicitly says to delete.
2. **Web↔desktop parity — POOR / HIGH RISK (your #1 concern).** Only the ~157-line root token sheet is shared. Below it, ~10,000 lines of component CSS are hand-maintained **twice**, 5+ screens are implemented **twice** with divergent markup *and vocabulary* ("Tools/Skills" on desktop vs "Connections/Abilities" on web), and **no test compares the two surfaces.** They are drifting today.
3. **Modularity — good boundaries, heavy duplication + god files.** Preload/package layering is exemplary, but `cloud/` has four 1,600–4,400-line files and feature logic/CSS is duplicated across apps.

The single highest-leverage move solves design *and* parity *and* modularity at once: **one screen = one shared `packages/ui` surface with one shared stylesheet, consumed by both apps.** That pattern already works for Projects/Approvals/Channels — it just hasn't been applied to Home, Knowledge graph, Team, Tools, Playbooks, or the component CSS.
